PROJECT OVERVIEW:
I want to build a Splitwise-like bill-splitting dApp with onchain settlement.
The app supports BOTH normal (public) payments and confidential payments
inspired by Fairblock’s confidentiality model.

The app must feel SIMPLE, CLEAN, and PROFESSIONAL — not clunky, not hacky.

CORE PRODUCT GOALS:
- Clear tracking of who PAID and who HAS NOT PAID
- Exact-payment enforcement (no partials, no manipulation)
- Optional confidentiality for payments
- Minimal UI, minimal features, maximum clarity
- Beginner-friendly architecture

────────────────────────
CORE FUNCTIONAL REQUIREMENTS
────────────────────────

BILLS:
1. A user can create a bill with:
   - Bill ID
   - Creator wallet address
   - List of participant wallet addresses
   - Exact amount owed per participant

2. For each participant, the system MUST track:
   - Has this participant paid? (YES / NO)
   - If paid:
     - Payment mode: NORMAL or CONFIDENTIAL
   - Payment timestamp (optional)

3. Bills are immutable after creation
   - Amounts cannot be changed
   - Participants cannot be changed

PAYMENTS:
4. Participants can settle their share by choosing:
   - Normal (public) payment
   - Confidential payment

5. EXACT PAYMENT RULE (VERY IMPORTANT):
   - Paying less than required → REJECT
   - Paying more than required → REJECT
   - Partial payments are NOT allowed
   - Contract must enforce correctness

CONFIDENTIAL PAYMENTS:
6. Confidential payments must:
   - Hide amount AND receiver onchain
   - Appear as a generic contract interaction
   - Validate correctness inside confidential execution
   - Mark participant as PAID without revealing amount publicly

7. The app UI must show:
   - Paid / Unpaid status
   - Payment mode (Normal / Confidential)
   - Amount ONLY if payment was normal

CHAIN SUPPORT:
8. App must support:
   - Base testnet
   - ARC testnet
   - Chain switching from frontend
   - Same contract logic deployed on both chains

────────────────────────
USER PROFILES & DATA STORAGE
────────────────────────

USER IDENTITY:
- Wallet address is the ONLY identity
- No passwords
- No usernames required

PROFILES:
- A lightweight user profile may exist
- Profile creation happens on first wallet connect
- Profile fields (all optional except wallet):
  - walletAddress
  - displayName
  - email (optional, for reminders)

DATA STORAGE RULES:
- Do NOT force a specific storage solution
- Propose multiple beginner-safe options:
  - Offchain database (e.g. free backend)
  - Decentralized storage (optional)
- Explain tradeoffs clearly
- Blockchain should NOT store profile data

EMAIL REMINDERS (OPTIONAL FEATURE):
- Users may opt-in to email reminders
- Reminders are handled OFFCHAIN
- Blockchain emits events only
- Backend listens to events and sends emails

PRIVACY RULE FOR EMAILS:
- If payment is confidential:
  - Do NOT include amount
  - Do NOT include recipient details
  - Only say “You have a confidential payment request”
